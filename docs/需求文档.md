## 1. 项目概述

### 1.1 背景

企业数据中台场景下，测试用例编写面临以下痛点：

| 挑战 | 传统方式 | Sisyphus 方案 |
|---|---|---|
| 需求散乱、格式各异 | 人工整合成大文档，耗时数天 | Requirement Card + 平台自动聚合 |
| 覆盖方向错了才发现 | 写完几十条用例才发现测错了方向 | **测试点显式评审**，先对齐「测什么」再生成 |
| 需求遗漏场景多 | 依赖个人经验，上线后才发现 | AI 主动诊断 + 苏格拉底式场景补全 |
| 需求变更影响不清 | 人工地毯式排查，极易遗漏 | Diff 引擎 + 用例影响自动标记 |
| 通用 AI 生成质量差 | 结果需大量二次修改 | RAG + 行业知识库 + 约束式生成 |
| 历史数据格式混乱 | 无法复用，每轮从头写 | 智能导入清洗 + 向量化历史用例库 |
| 跨迭代知识不传承 | 经验断层，质量无法累积 | 执行结果回流 + 持续学习飞轮 |

### 1.2 目标效果

- ⚡ 单模块用例生成时间：**天级 → 小时级**（效率提升 70%+）
- 🎯 场景覆盖率：**达到人工编写的 90% 以上**
- 🔁 测试点评审通过后再生成：**大幅减少返工率**（方向错误在前期拦截）
- 🔍 需求变更影响：**秒级定位**受影响用例
- 📈 跨迭代质量提升：执行结果回流，每轮生成质量持续优化

### 1.3 产品定位

Sisyphus Case Platform 是面向**企业数据中台场景**的 AI 功能测试用例智能生成平台，支持子产品多迭代并行研发模式，覆盖从需求录入、健康诊断、测试点分析、用例生成、变更追踪到执行回流的**完整测试生命周期**。

## 2. 核心理念

### 2.1 显式拆分「测什么」与「怎么测」

测试用例生成的本质包含两个截然不同的决策：

| 决策 | 核心问题 | 评审方 | 错误代价 |
|---|---|---|---|
| **测试点**（场景地图） | **测什么？** 覆盖哪些场景 | 测试工程师 + 产品经理 | 低（改一行描述） |
| **用例生成** | **怎么测？** 具体步骤和数据 | 测试工程师 | 高（可能重写数十条用例） |

如果跳过测试点直接生成用例，发现覆盖方向错了需要推倒重来一批用例。而在测试点阶段对齐方向，一条测试点描述改一下，后续 5 条用例可自动重新生成。

### 2.2 评审节点前置拦截

```
需求卡片确认 → 健康诊断 → 场景补全 → 【测试点评审确认】→ 用例生成 → 执行回流
                                           ↑
                                     关键评审节点
                                     方向错误在此拦截
```

**设计原则**：测试点的粒度和语言对产品经理/业务方友好——他们看不懂用例的步骤格式，但能快速评审「这 18 个测试点覆盖了你的需求吗」，是最高效的评审层级。

### 2.3 质量飞轮

```
[需求录入] → [健康诊断] → [测试点确认] → [用例生成] → [执行] → [结果回流]
      ↑_________________________________________________________________↓
              每轮迭代：生成质量↑  测试点准确率↑  人工修改量↓
```

## 4. 核心功能模块

### 4.1 M00 — 子产品/迭代/需求三级管理

**优先级**: P0

**层级结构**：

```
Platform
  └── 子产品（离线开发 / 实时开发 / 数据资产 / 数据治理 / BI分析...）
        └── 迭代（2025-Q2-Sprint3）
              └── 需求（REQ-OFFLINE-089，约1页PRD）
                    ├── 测试点列表（场景地图节点）
                    ├── 关联用例列表（可溯源到测试点）
                    └── 变更历史（Diff记录）
```

**核心功能**：

- **子产品管理**：创建/编辑子产品，设置子产品属性（名称、slug、描述）
- **迭代周期管理**：迭代创建、起止日期、状态管理（active/completed）
- **需求卡片编辑器**：
  - Notion 风格块编辑器（基于 TipTap/ProseMirror）
  - Markdown + YAML frontmatter 底层存储
  - `/` 唤起：标题/列表/表格/图片/分割线
  - 图片粘贴自动上传归档
  - AI 辅助生成验收标准草稿
  - 版本快照（每次确认需求自动创建）

**三级结构 vs 大文档方式对比**：

| 维度 | 大文档方式 | 三级结构方式 |
|---|---|---|
| 录入成本 | 需人工整合，格式不统一 | 一页 PRD 即一张卡片，独立录入 |
| 测试点粒度 | 大文档测试点容易混淆 | 每条需求独立生成测试点，粒度清晰 |
| 变更追踪 | 无法定位到单条需求 | Diff 精准到需求节点级别 |
| AI 生成 | 大文档超出上下文窗口 | 按迭代自动聚合，智能分组送入 |

### 4.2 M01 — 文档解析引擎（UDA 层）

**优先级**: P0

> **核心原则**：平台内部永远不存原始文件格式，只存结构化内容。格式处理只发生在输入层和输出层。

**UDA（Universal Document Abstraction）处理流程**：

```
任何格式输入               内部统一数据模型 (AST)           任何格式输出
──────────────            ──────────────────────          ─────────────
.docx        ──┐          ┌──────────────────┐            → .md
.pdf         ──┤          │  DocumentNode    │            → .docx
.md          ──┤  解析层→  │  ├ metadata      │  渲染层──→  → .pdf
.csv         ──┤          │  ├ sections[]     │            → .xlsx
图片(本地)   ──┤          │  │  ├ text         │            → XMind
网络图片URL  ──┤          │  │  ├ images[]     │            → Jira/Xray
Confluence链接 ┘          │  └ rules[]        │
                          └──────────────────┘
```

**输入格式支持**：

| 输入类型 | 解析方式 | 特殊处理 |
|---|---|---|
| `.docx / .doc` | python-docx | 保留标题层级、表格、列表 |
| `.pdf`（文本版） | pdfplumber/PyMuPDF | 保留段落结构、页码映射 |
| `.pdf`（扫描版） | PaddleOCR | 置信度 < 0.8 时标记人工复核 |
| `.md` | Marked.js | 提取 frontmatter YAML + 章节树 |
| `.xlsx / .csv` | OpenPyXL / pandas | 字段映射 + 枚举值提取 |
| 图片（本地） | PaddleOCR + GPT-4 Vision | 识别流程图节点、界面标注 |
| 网络图片 URL | 主动抓取→存 MinIO→OCR | 替换内部 URL，防外链失效 |
| Confluence/飞书链接 | 调用对应 API | 需配置企业系统 Token |

**导出格式矩阵**：

| 导出对象 | 支持格式 |
|---|---|
| 需求文档 | `.md` / `.docx` / `.pdf` |
| 测试用例集 | `.xlsx` / `.md` / `.docx` / `.pdf` / XMind |
| 测试点 & 场景地图 | `.png`（可视化）/ `.md`（结构列表）/ `.json` |
| 需求健康报告 | `.pdf` / `.md` |
| 迭代测试计划 | `.docx` / `.pdf` |
| 需求覆盖度矩阵 | `.xlsx` / `.png` |
| Diff 报告 | `.md` / `.html`（带高亮） |

### 4.3 M02 — 历史数据导入与清洗

**优先级**: P0

**双轨处理策略**：

```
导入文件（CSV历史用例 / MD历史文档 / Word / PDF）
  │
  ├── [底层静默处理] 字段映射 / 图片归档 / 基础去重 / 空值填充 / 向量化
  │
  └── [导入健康报告] 展示需要人工决策的部分

✅ 自动处理成功   892 条  (72%)
⚠️ 需要人工确认   298 条  (24%)  ← AI做决策，人做审核
❌ 无法解析         50 条   (4%)
```

**人工确认类型**：

| 情况 | 处理方式 |
|---|---|
| 字段语义模糊（"测试点"是标题还是步骤？） | 展示两种解析结果，用户选择 |
| 图片 URL 已失效（404） | 标记"图片丢失"，支持用户重新上传 |
| 步骤全部塞在一个单元格 | AI 拆分结果 + 原始内容并排，用户确认 |
| 优先级值非标准（"重要"/"紧急"） | 展示映射关系，用户确认 |
| 疑似重复（标题相似度 > 85%） | 并排展示，用户选择合并/保留/删除 |

### 4.4 M03 — 需求健康诊断

**优先级**: P0

需求健康诊断系统是平台的核心 AI 能力之一，通过多轮 AI 对话识别需求文档中的遗漏和风险。

#### 4.4.1 六类遗漏识别

| 遗漏类型 | AI 追问示例 |
|---|---|
| **异常路径缺失** | "§3.2 描述了同步成功流程，但未提及网络中断后的恢复策略" |
| **边界值未定义** | "`timeout` 为 Integer，未说明最大值、0值、负数的处理逻辑" |
| **权限场景缺失** | "数据源删除功能未说明：普通用户是否有删除权限？" |
| **并发竞争未说明** | "两个用户同时修改同一任务配置时，是否有锁机制？" |
| **状态流转不完整** | "任务状态有「执行中→成功/失败」，但未提及「暂停」和「取消」" |
| **跨模块关联缺口** | "§3.4 依赖 §2.2，但未说明数据源断开时调度任务的行为" |

#### 4.4.2 多轮深挖 — 三层终止架构

```
第一轮：广度扫描 ── AI 列出所有疑似遗漏，用户快速标记优先级（~10分钟）
第二轮：深度聚焦 ── 逐条深挖「需讨论」项，每项最多 3 层追问
第三轮：交叉验证 ── 检查新增场景一致性，识别衍生缺口
终止条件：红色/灰色节点归零，或用户点击「完成」
```

#### 4.4.3 行业必问清单（6 大类 32 条）

开箱即用的内置检查清单，覆盖企业数据中台常见场景：

| 类别 | 条数 | 触发条件 |
|---|---|---|
| 🔁 幂等性类 | 6 条 | 任何写入/更新操作 |
| ⏰ 时区与时间类 | 5 条 | 任何含时间戳的调度/记录 |
| 📦 大数据量与性能类 | 5 条 | 任何查询/导出/同步操作 |
| 🔀 状态流转与并发类 | 6 条 | 任何有状态机或共享资源的功能 |
| 💥 数据一致性与容错类 | 5 条 | 任何跨系统/跨表操作 |
| 🔒 权限与安全类 | 5 条 | 任何涉及数据访问或操作的功能 |

示例（幂等性类）：
1. 重复执行此操作，结果是否一致？重复写入如何处理（覆盖/报错/忽略）？
2. 网络超时导致客户端重试时，服务端是否会产生重复数据？
3. 是否有唯一键/幂等键机制保障？
4. 批量操作中部分成功时，重试是全量还是仅重试失败部分？

#### 4.4.4 诊断产出物

- **需求健康报告**：6 类风险汇总、每项风险的严重程度和建议
- **场景补全会话记录**：多轮对话历史，可追溯
- **补全测试点**：诊断过程中识别的遗漏场景自动作为待确认测试点

### 4.5 M04 — 测试点分析 & 场景地图

**优先级**: P0

> **设计决策**：测试点 = 场景地图节点，两者统一，不另起炉灶。

#### 4.5.1 测试点生成

AI 基于需求文档 + 健康诊断结果自动生成测试点草稿，每个测试点遵循粒度规范：

```
❌ 太粗（等于需求描述，没有指导意义）：
   - 验证数据同步功能正常

❌ 太细（等于用例步骤，跳过了抽象层）：
   - 验证输入 Host=192.168.1.100, Port=3306 时连接成功返回"连接成功"

✅ 合适粒度（一个测试点对应 2~5 条用例）：
   - 验证 MySQL/Oracle/PostgreSQL 数据源 JDBC 连接参数配置的有效性
   - 验证连接超时参数（timeout）的边界值处理行为
   - 验证连接失败时系统返回的错误提示可读性与准确性
```

#### 4.5.2 测试点分类与颜色编码

| 颜色 | 含义 | 来源 | 状态 |
|---|---|---|---|
| 🟢 绿色 | 需求文档已覆盖 | `document` | `covered` |
| 🟡 黄色 | AI 补充识别 | `ai_detected` | `supplemented` |
| 🔴 红色 | 待处理遗漏 | `ai_detected` | `missing` |
| ⚪ 灰色 | 待确认 | `ai_detected` | `pending` |

#### 4.5.3 双视图展示

- **📋 测试点列表视图**：按场景类型分组（正常/边界/异常/并发/权限），显示预计用例数，适合评审确认
- **🌳 场景地图可视化**（React Flow）：树形展示覆盖全貌，适合管理层汇报

#### 4.5.4 用例与测试点的双向溯源

```
TC-089-003  ←→  测试点: "连接超时参数边界值处理"  ←→  REQ-089 §2.3
TC-089-004  ←→  测试点: "连接超时参数边界值处理"  ←→  REQ-089 §2.3

需求变更时：
  §2.3 变更 → 自动标记 "连接超时参数边界值处理" 测试点受影响
           → 自动标记 TC-089-003/004 需要复核
```

### 4.7 M06 — 用例管理中心

**优先级**: P0

**核心能力**：

- **全链路溯源**：用例 → 测试点 → 需求 → 迭代

- **筛选与搜索**：按优先级/类型/状态/需求筛选

### 4.8 M07 — 需求 Diff & 变更影响分析

**优先级**: P0

#### 4.8.1 两阶段 Diff 引擎

```python
class DiffEngine:
    """两阶段 Diff：文本级（快速）+ 语义级（准确）"""

    def compute_diff(self, v_old, v_new):
        # 第一阶段：Myers 算法文本级 Diff
        text_diff = self._myers_diff(v_old, v_new)
        # 第二阶段：LLM 判断业务影响
        # 防止"30秒改10秒"被误判为无关紧要的措辞变化
        semantic_impact = self._llm_impact_analysis(text_diff)
        return DiffResult(changed_nodes=text_diff.nodes,
                          impact_level=semantic_impact)
```

#### 4.8.2 影响分析产出

```
影响分析
  🔴 测试点需重新生成    2 个  → 「超时边界值处理」「自定义间隔配置」
  🔴 用例需要重写        4 条  → TC-089-003、TC-089-007...
  🟡 用例需要复核       11 条  → TC-089-001、TC-089-012...
  🟢 不受影响          231 条
```

#### 4.8.3 关键功能

- 版本对比视图（并排 Diff）
- 变更范围自动识别
- 受影响测试点自动标记
- 受影响用例自动标记（必须重写/建议复核）
- 一键重新生成受影响用例

## 5. 核心工作流程

### 5.1 主流程

```
① 需求录入
   ├── Requirement Card 编辑器录入新需求
   ├── 历史数据导入（CSV/MD/Word/PDF）
   └── 确认需求卡片，创建版本快照
         │
         ▼
② 需求健康诊断
   ├── AI 自动扫描 6 类遗漏风险
   ├── 生成需求健康报告
   └── 苏格拉底式场景补全对话（三层终止架构）
         │
         ▼
③ 测试点分析与确认  ← 【关键评审节点】
   ├── AI 生成测试点草稿（基于需求 + 诊断结果）
   ├── 用户评审：增删改测试点
   ├── 可与 PM/业务方共同评审（分享评审链接）
   └── 确认后锁定：「测什么」已定
         │
         ▼
④ 用例生成
   ├── 推荐：测试点驱动批量生成
   ├── 可选：文档驱动 / 对话引导 / 模板填充
   ├── 每个测试点 → 2~5 条详细用例
   └── 流式输出，实时预览
         │
         ▼
⑤ 用例审核与导出
   ├── 用例管理中心 CRUD
   ├── 全链路溯源：用例 → 测试点 → 需求 → 迭代
   └── 多格式导出 / Jira 集成
         │
         ▼
⑥ 执行与回流
   ├── 执行结果上报（手动 / Jira 回写）
   ├── 失败用例标记、高质量用例进入优质库
   └── 回流数据驱动下轮生成质量提升
```

### 5.2 变更追踪流程

```
需求变更（编辑需求卡片）
    │
    ▼
自动创建新版本快照
    │
    ▼
Diff 引擎计算（文本级 Myers + 语义级 LLM）
    │
    ▼
影响分析报告
    ├── 受影响测试点标记
    ├── 受影响用例标记（重写 / 复核）
    └── 通知相关测试工程师
    │
    ▼
一键重新生成受影响用例
```

### 5.3 执行结果回流机制

| 执行结果 | 回流处理 | 对后续生成的影响 |
|---|---|---|
| 用例执行失败 | 记录失败原因 | AI 自动规避同类描述缺陷 |
| 用例被删除 | 标记删除原因 | 降低此类测试点的生成权重 |
| 高质量通过用例（评分≥4.5） | 进入优质用例库 | RAG 优先引用，作为 Few-shot |
| 用例人工修改记录 | 记录修改字段和内容 | 分析模式，优化 Prompt 模板 |
| 测试点被大幅修改 | 记录修改方向和原因 | 优化测试点生成策略 |

### 6.3 软删除规范

所有核心业务表均包含 `deleted_at TIMESTAMPTZ DEFAULT NULL` 字段：

- 删除操作：`UPDATE SET deleted_at = now()`
- 查询过滤：`WHERE deleted_at IS NULL`
- 数据恢复：通过回收站（M21）将 `deleted_at` 置 NULL
- 定期清理：超过保留期限的软删除数据永久清除
