# 设计文档：历史 CSV 用例数据批量清洗

**日期：** 2026-03-21  
**状态：** 已批准

## 问题陈述

`清洗后数据/` 目录下存有 228 个历史测试用例 CSV 文件，每个文件含 9 列：

```
用例编号, 用例标题, 所属模块, 相关需求, 前置条件, 步骤, 预期结果, 优先级, 用例类型
```

其中 `相关需求` 的值等同于文件名（冗余），`用例编号` 和 `用例类型` 也不再需要。
目标是精简为平台导入所需的 6 列，减少噪声，便于后续 RAG 知识库摄取。

## 目标

- 保留列（顺序固定）：`所属模块, 用例标题, 前置条件, 步骤, 预期结果, 优先级`
- 删除列：`用例编号, 相关需求, 用例类型`
- 原地覆盖所有文件（Git 可回溯历史）
- 编码统一为 `utf-8-sig`（BOM，保证 Excel 正常显示中文）

## 方案选择

选择**方案 A：纯 Python `csv` 模块**。

理由：
- 零额外依赖，标准库即可
- `csv.DictReader` / `DictWriter` 正确处理含逗号、换行符的引用字段
- 所有 228 个文件列结构一致，逻辑简单可验证

放弃方案 C（shell awk/cut）：用例步骤/预期结果字段含内部逗号，awk 按列截取会破坏数据。

## 实现设计

### 脚本位置

`scripts/clean_csv_columns.py`（项目根级脚本，不进入后端包）

### 核心逻辑

```
for each *.csv in 清洗后数据/**:
    read with csv.DictReader(utf-8-sig)
    write back with csv.DictWriter(utf-8-sig)
        fieldnames = [所属模块, 用例标题, 前置条件, 步骤, 预期结果, 优先级]
    print ✓/✗ per file
print summary: total / ok / error
```

### 错误处理

- 单文件出错（编码异常、列缺失）：打印错误，跳过该文件，继续处理其余文件
- 写入前先完整读取内存，防止写入中途崩溃导致文件损坏（原子写入：写临时文件 → rename）

### 成功验证

运行后抽查 3~5 个文件，确认：
1. 列头为 6 列且顺序正确
2. 行数与处理前一致
3. 含特殊字符（逗号/换行）的字段未被截断

## 约束

- Python 3.x 标准库，无需安装额外依赖
- 脚本幂等：重复运行不改变已清洗文件的结果
