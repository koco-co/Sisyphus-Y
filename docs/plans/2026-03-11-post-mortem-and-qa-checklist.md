## 一、事故根因分析

### 1.1 为什么"自测通过"≠"客户可用"

本次交付的所有测试都存在**根本性局限**，导致虚假的"全部通过"信号：

| 测试类型 | 表面覆盖 | 实际盲区 |
|---------|---------|---------|
| 后端单元测试（80 个文件） | 用 `AsyncMock` / `patch` 把数据库和 AI 全部模拟掉 | **真实 PostgreSQL 查询、真实 AI API 调用从未执行** |
| P7 接口测试（"20/20 通过"） | 各端点返回 HTTP 200，不崩溃 | **业务逻辑正确性、数据写入、SSE 流式输出未验证** |
| P8 浏览器测试（"13 页渲染正常"） | 页面不白屏 | **用户实际操作流程（需求→诊断→测试点→用例）从未走通** |

> **核心教训：AI 写代码 + AI 写测试 = 自己出题自己答，天然存在覆盖盲区。**
> 测试覆盖的是"AI 认为会发生什么"，不是"用户实际怎么操作"。

### 1.2 发现的高风险问题清单

#### P0：AI 功能可能完全失效

`.env` 文件配置了 `LLM_PROVIDER=openai`，但 `app/core/config.py` 默认 `llm_provider=dashscope`。

- 若客户环境**没有配置 OpenAI API Key**，或**网络无法访问 OpenAI**，则诊断、场景地图、用例生成全部报错
- 这意味着整个平台的核心价值（AI 驱动的用例生成）直接失效

**修复**：交付前必须明确告知客户需要配置哪个 provider 的 Key，并在客户环境中实际触发一次 AI 调用验证。

#### P1：核心链路从未端到端验证

「需求录入 → 健康诊断 → 测试点确认 → 用例生成」这条链路：
- 每个模块单独测试了接口返回 200
- **模块间的数据流转从未整体验证**
- 例如：诊断报告生成后，前端的测试点页面能否正确读取并展示？

#### P2：`progress.json` 数据矛盾

P1–P8 各 phase 的 `status` 字段全部为 `"pending"`，但内部 task 都是 `"done"`——进度追踪系统本身有 bug，导致"270/270 完成"的数字不可靠。

#### P3：基础设施依赖未验证

平台需要 PostgreSQL + Redis + Qdrant + MinIO **同时正常运行**。这些服务的连通性在客户环境中**没有交付前验证脚本**，任何一个服务不可用都会导致连锁报错。

## 三、交付前验收清单（下次使用）

> 以下清单的每一项必须在真实环境（非本地 mock）中实际执行并通过，才能交付。

### ✅ 3.1 基础设施就绪检查

```bash
# 所有服务必须 healthy
docker compose -f docker/docker-compose.yml ps
# 期望输出：postgres/redis/qdrant/minio 全部 Up (healthy)

# 数据库迁移必须最新
cd backend && uv run alembic current
# 期望输出：head（无 pending migration）

# 检查健康端点
curl -f http://localhost:8000/health || echo "FAIL: 后端未启动"
curl -f http://localhost:3000/api/progress || echo "FAIL: 前端未启动"
```

### ✅ 3.2 AI 功能验证（最关键）

```bash
# 明确当前使用的 LLM provider
grep LLM_PROVIDER .env

# 手动触发一次 AI 调用（以诊断为例）
# 1. 在前端创建一个需求
# 2. 进入诊断页面，点击"开始诊断"
# 3. 观察 SSE 流是否有内容输出（不是空白/报错）
# 4. 等待诊断完成，检查报告中是否有风险项
```

**如果 AI 无输出：立即检查**
```bash
# 查看后端日志中的 LLM 调用错误
docker logs sisyphus_backend 2>&1 | grep -E "LLM|Error|401|timeout"
```

### ✅ 3.3 核心端到端流程（必须手工走一遍）

按顺序操作，每步检查数据是否真实入库：

- [ ] 创建产品 → 创建迭代 → 创建需求（填写有实际内容的需求描述）
- [ ] 进入诊断页面 → 触发 AI 诊断 → 等待报告生成（有风险项出现）
- [ ] 进入场景地图页面 → 触发 AI 生成测试点 → 确认测试点（状态变绿）
- [ ] 进入工作台 → 触发用例生成 → 查看用例卡片有 steps 内容
- [ ] 进入用例管理页面 → 确认用例已持久化（刷新页面后仍然存在）
- [ ] 导出用例 → 验证 Excel/JSON 文件内容完整

**以上任何一步失败 = 不能交付**
