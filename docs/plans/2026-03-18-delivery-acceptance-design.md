# Sisyphus-Y 交付验收与体验整改设计

> 日期：2026-03-18
> 范围：全站交付验收、问题闭环、文档收敛、主流程体验优化
> 原则：只记录当前代码中已实现能力；验收、问题、修复验证统一纳入同一套进度模型

## 目标

本轮不是单点修 Bug，而是把 Sisyphus-Y 拉回到一个可交付、可验收、可持续收敛的状态。

目标拆成四个层面：

1. 全站所有现有入口都必须被验收，不允许只测主流程。
2. progress.json 需要同时承载验收任务、发现的问题、修复验证，让前端进度大盘展示完整交付过程。
3. 对 test_plan、占位按钮、难理解入口、重复文档、历史残留模块，逐项给出保留、精简、隐藏、重构或删除建议，而不是拍脑袋统一下线。
4. 优先修复主流程中的用例生成体验，尤其是思考过程、加载反馈、错误态、流式状态可信度。

## 现状摘要

### 已确认可复用基础

- 前端已存在全局进度大盘，读取根目录 progress.json 并展示为 phase -> module -> task 的层级视图。
- 后端 generation 已具备 SSE、thinking_content 持久化和会话落库能力。
- 设置页 AI 模型配置已支持 openrouter、自定义 OpenAI-compatible provider 的前端 provider 清单。
- 工作台生成面板已存在 ThinkingStream、StreamCursor、错误态和重试按钮组件。

### 已确认的核心不一致

- 文档声称顶栏是 9 菜单版，但前端当前仍暴露 test-plan 入口。
- README、CLAUDE、AGENTS、copilot-instructions 存在大量重复、过期或超前描述。
- progress.json 当前更像传统验收清单，还没有显式区分“问题发现”和“修复验证”。
- 根目录和 frontend 同时存在 Node 生态目录，需区分“工具链目录”与“产品运行目录”，不能简单粗暴认定为脏目录后删掉。

## 设计决策

### 1. 进度模型采用三轨合一

progress.json 中所有条目统一仍放在 tasks 数组，但按 category 和 id 规则拆成三类：

- 验收任务：覆盖页面、模块、按钮、接口、兼容性与文档一致性检查。
- 问题条目：记录体验问题、交互问题、后端逻辑问题、文档问题、结构问题。
- 修复验证：对应某个问题条目的回归验证，必须带浏览器或接口证据。

这样做的原因：

- 不需要先改前端大盘组件结构，就能在现有展示模型内承载完整交付过程。
- 允许按模块滚动推进，一个页面验收时发现的问题可以立即入表，不需要另开系统。
- 与当前 progress API 的 category 分组逻辑兼容，只需增强映射与命名规则。

### 2. 先验收现状，再做删留改建议

对 test_plan、通知、全局搜索 mock、设置页集成面板、帮助入口、知识库交互等内容，先基于代码和实测结果做“事实登记”，再给出建议：

- keep：功能完整且心智清晰
- simplify：保留能力但隐藏复杂参数，补默认值
- hide：能力保留但暂时移出主导航或弱化入口
- refactor：功能有价值但结构混乱，需要重构后保留
- remove：价值低且维护成本高，建议删除

### 3. 文档治理采用“一主多从”

建议文档分层：

- README：对外总览，只写当前已实现能力与启动方式
- CLAUDE / copilot-instructions：开发约束与仓库事实
- AGENTS：缩成索引和执行说明，不再复制整份 CLAUDE
- CHANGELOG：仅记录已完成改动
- docs/功能实现文档：作为“当前代码能力全量说明”的主事实文档

### 4. 主流程体验优先级最高

本轮体验整改优先级：

1. 工作台用例生成流式反馈可信度
2. AI 配置与模型选择体验
3. markdown 渲染、图片回显、文件预览
4. 分析台、Diff、用例库的入口和选择器可理解性
5. 非主流程模块的收口与弱化

## 验收阶段划分

### Phase A：环境与结构基线

- 启动前后端、基础设施、健康检查
- 识别根目录工具链目录、frontend 运行目录、遗留目录
- 确认 process.json / progress.json / docs/plans 的治理方式

### Phase B：全站浏览器验收

- 仪表盘
- 分析台
- 工作台
- Diff
- 用例库
- 模板库
- 知识库
- 回收站
- 设置
- 当前仍暴露的 test-plan

### Phase C：问题闭环与回归

- 每个问题都必须挂到 progress.json
- 每个修复都必须有对应验证条目
- accepted 只能在真实浏览器证据、关键 lint/typecheck、关键主流程回归都完成后置为 true

## 风险与约束

- 当前工作树已存在大量未提交修改，实施期间必须只增量修改，不回滚用户现有改动。
- 浏览器实测是硬门槛，因此需要保证本地前后端可启动；仅靠静态读代码不能宣称验收通过。
- 大量 markdown 文档存在事实漂移，更新时必须以当前代码实现为准，不能把计划态写成已上线。

## 本轮实施顺序

1. 先补设计文档与实施计划。
2. 改 progress.json 与前端 progress 展示的基础模型。
3. 启动应用并做第一轮全站验收。
4. 先修主流程体验问题，再处理结构与文档问题。
5. 最后更新 CHANGELOG 与验收证据。
