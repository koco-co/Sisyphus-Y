# UX 增强设计文档

**日期**：2026-03-21
**范围**：全平台 UX 改进 —— 流程引导 + 交互增强 + 文案重写 + Kimi K2 验证
**背景**：全量功能测试后，主链路（需求 → 分析 → 测试点 → 用例）存在跨模块割裂感、AI 等待体验差、操作反馈弱、文案生硬等问题。

---

## 一、问题诊断

| 问题类型 | 具体表现 |
|---------|---------|
| 流程割裂 | 主链路 4 步横跨多个路由，每步完成后用户不知道下一步在哪 |
| 等待体验差 | AI 生成时仅有光标动画，不知道是在跑还是卡死 |
| 跳转生硬 | 页面切换无过渡，体感像"闪现" |
| 操作反馈弱 | 按钮点击无即时响应，Toast 风格不统一 |
| 文案生硬 | 空状态用"暂无数据"，错误用技术性描述，无引导性 |

---

## 二、设计方案：渐进增强全家桶（三线并行）

### 维度 1：流程引导体系

#### 1.1 WorkflowStepper 组件

在分析台和工作台顶部持久化显示 4 步进度条：

```
① 需求准备  →  ② AI 分析  →  ③ 确认测试点  →  ④ 生成用例
  ✓ 已完成      ● 当前步骤     ○ 待进行         ○ 待进行
```

**状态驱动逻辑**（由需求字段驱动）：

| 需求状态字段 | 对应步骤激活 |
|------------|------------|
| `diagnosis_status = null` | 步骤 2 高亮，显示"开始分析"CTA |
| `diagnosis_status = completed` | 步骤 2 完成，步骤 3 高亮 |
| `scene_map_status = confirmed` | 步骤 3 完成，步骤 4 高亮 |
| 用例已生成 | 步骤 4 完成，全链路收尾 |

**组件规格**：
- 路径：`components/workflow/WorkflowStepper.tsx`
- 在 `analysis/[id]` 和 `workbench/[id]` 页面引入
- 步骤点击可跳转到对应路由（已完成步骤可回跳）

#### 1.2 SmartNextCard 组件

每当 AI SSE 流完成后，在结果区底部浮出引导卡片：

```
┌──────────────────────────────────────────────────┐
│  AI 分析完成，识别到 5 个高风险点                  │
│  建议下一步：确认 12 个测试点，再进入工作台生成用例 │
│                           [前往确认测试点 →]       │
└──────────────────────────────────────────────────┘
```

**规格**：
- 路径：`components/workflow/SmartNextCard.tsx`
- 触发时机：SSE `done` 事件后 500ms 延迟显示
- 内容由父组件传入（根据当前步骤决定文案和跳转路径）
- 用户点击 CTA 或手动关闭后隐藏

#### 1.3 空状态引导

所有模块空状态统一为"说明 + 行动按钮"结构，禁止纯文字"暂无数据"。

### 维度 2：交互增强体系

#### 2.1 AI 流式输出三阶段状态机

```
[思考中] → [生成中] → [整理中] → [完成]
```

| 阶段 | 触发条件 | 视觉表现 | 文案 |
|------|---------|---------|------|
| 思考中 | SSE 开始，`thinking` 事件期间 | 脉冲动画圆点 | "正在理解需求结构..." |
| 生成中 | `content` 事件期间 | 文字流式渲染 + 光标 | 实时内容 |
| 整理中 | 内容完成，等待 `done` | 淡入淡出光标 | "正在整理分析结果..." |
| 完成 | `done` 事件后 | 结果静止 + SmartNextCard | — |

#### 2.2 页面过渡动画

在 `(main)/layout.tsx` 加入 `PageTransition` 包装层：
- 进入动画：`opacity: 0→1` + `translateY(4px→0)`，时长 `150ms ease-out`
- 退出动画：`opacity: 1→0`，时长 `100ms`
- 实现方式：CSS `@keyframes` + Next.js `usePathname` 触发重渲染

#### 2.3 操作反馈规范

| 场景 | 反馈方式 | 持续时间 |
|------|---------|---------|
| 异步操作触发 | 按钮 `loading` 态（Loader2 + 文字切换） | 直到完成 |
| 操作成功 | `toast.success`，右上角滑入 | 3s 自动消失 |
| 操作失败 | `toast.error` + 可操作建议 | 5s / 手动关闭 |
| AI 任务完成 | SmartNextCard（非 Toast） | 用户手动关闭 |

#### 2.4 骨架屏统一

以下区域加载时使用骨架屏（替代空白等待）：
- 分析台需求列表
- 用例库用例列表
- 工作台测试点列表
- 仪表盘统计卡片

骨架屏组件统一使用 `components/ui/TableSkeleton.tsx`（已存在），扩展卡片形态。

### 维度 3：文案全量重写

#### 文案原则

1. **动词开头**：按钮/CTA 一律"动词 + 对象"
2. **有下一步**：空状态必须包含行动引导
3. **拟人但克制**：AI 说话像专业同事
4. **错误可行动**：报错必须包含"你可以…"

#### 关键文案对照表

| 位置 | 现有文案 | 重写后 |
|------|---------|--------|
| 分析台空状态 | 暂无分析报告 | 还没有分析过这个需求，点击「开始分析」让 AI 识别潜在风险 |
| AI 启动提示 | 正在加载... | 正在理解需求结构，通常需要 10–30 秒 |
| AI 思考阶段 | （无） | 正在从 6 个维度扫描需求风险... |
| 测试点空状态 | 暂无测试点 | AI 还没有生成测试点，先完成需求分析再来这里 |
| 工作台空状态 | 请先选择需求 | 从左侧选择一个需求，AI 会根据已确认的测试点生成用例 |
| 用例生成中 | 生成中... | 正在为「${测试点名}」编写测试步骤... |
| 操作成功（通用） | 操作成功 | 已确认 ${n} 个测试点 |
| 删除确认 | 确认删除吗？ | 删除后可在回收站找回，30 天内有效 |
| 知识库空状态 | 暂无文档 | 上传测试规范或历史用例，AI 生成时会自动参考这些知识 |
| Prompt Tab 空状态 | 暂无 Prompt 配置 | 所有 AI 分析模块的指令都在这里，点击任意模块可查看或调整 |
| 回收站空状态 | 暂无记录 | 回收站是空的 |
| 网络错误 | 请求失败（500） | 服务暂时无响应，你可以刷新页面或稍后重试 |
| Diff 空状态 | 输入需求 ID 并选择版本范围 | 选择同一需求的两个版本，AI 将找出变更并评估对测试用例的影响 |
| 需求列表空状态 | 暂无需求 | 还没有需求，从录入或上传文档开始 |
| 用例库空状态 | 暂无用例 | 用例库是空的，先去工作台生成第一批用例 |

#### 文案覆盖范围

需重写的组件（约 30 个）按优先级分为：
- **P0**（主链路直接可见）：分析台、工作台、用例库
- **P1**（辅助模块）：知识库、模板库、回收站、设置
- **P2**（边缘场景）：错误边界、网络错误、权限提示

### 维度 4：Kimi K2 模型验证

**目标**：在设置页切换默认模型为硅基流动 / Kimi K2，用真实需求文档跑完整分析链路，评估输出质量。

**验证步骤**：
1. 设置 → AI 模型配置 → 将硅基流动-Kimi K2 设为默认
2. 用 `Story-15602.md` 跑分析（需求分析 → 场景地图 → 用例生成）
3. 对比 GLM-4-Flash 输出，评估：中文流畅度、分析深度、用例格式规范性
4. 根据结果决定是否需要调整对应模块的 Prompt

---

## 四、技术约束

- 新增组件路径遵循现有目录规范（`components/workflow/`、`components/ui/`）
- 页面过渡动画不得阻塞首屏渲染，150ms 为硬性上限
- 文案修改只改中文字符串，不动代码标识符
- Kimi K2 验证完成后，模型选择作为配置保留，不硬编码
