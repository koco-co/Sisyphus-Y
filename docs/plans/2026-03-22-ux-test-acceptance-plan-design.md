# Sisyphus-Y 全量 UX 测试验收计划 — 设计文档

**日期**：2026-03-22  
**版本**：1.0  
**文档类型**：设计文档（Design Doc）  
**目标**：通过 Claude in Chrome MCP 对平台所有功能进行自动化测试，从用户视角发现 UX 问题，输出带改进理由的综合 UX 审计报告。

---

## 1. 背景与目标

### 1.1 当前状态

Sisyphus-Y 平台（v2.1.0）已完成 75/87 个验收任务（86% 通过率），主要功能模块已实现。当前缺少一次系统性的、从真实用户视角出发的 UX 全量测试，以发现：

- 哪些功能不合理或需要重新设计？
- 哪些功能存在整合机会（避免重复/碎片化）？
- 哪些交互细节需要优化（表单、状态反馈、错误处理）？
- 整体用户旅程是否顺滑？

### 1.2 测试目标

1. **功能可用性**：每个功能是否按预期工作
2. **UX 合理性**：交互设计是否符合用户心智模型
3. **流程顺滑度**：跨模块衔接是否自然
4. **错误友好性**：边界情况和异常状态的处理

### 1.3 测试方式

- **工具**：Claude Code + Claude in Chrome MCP（`mcp__claude-code-chrome__*` 系列工具）
- **目标地址**：`http://localhost:3000`
- **测试视角**：一名新用户第一次使用平台的完整体验

## 4. 问题记录格式

每个发现的问题按以下模板记录到输出报告：

```markdown
### [UX-XXX] 问题标题

- **位置**：页面路径 / 组件名
- **触发步骤**：场景编号 → 步骤编号（如 S2-07）
- **问题描述**：（从用户视角描述看到/遇到的情况）
- **用户影响**：（对用户体验/效率的影响程度）
- **改进建议**：（具体的改进方案）
- **改进理由**：（为什么这么改——参考设计原则、竞品、用户认知）
- **优先级**：高 / 中 / 低
- **分类**：可选项—信息架构 / 交互设计 / 视觉一致性 / 功能整合 / 错误处理 / 性能感知
```

## 7. 设计决策说明

| 决策 | 理由 |
|------|------|
| 用户视角而非功能列表 | 功能是否存在不代表体验好；用户旅程能发现跨模块衔接问题 |
| 4 场景而非逐页测试 | 减少重复覆盖，每场景有明确的用户故事背景 |
| 问题带改进理由 | 改进理由让开发者理解"为什么"，增加执行意愿 |
| 输出到 docs/ 而非 issues | 便于长期追踪和分享，可转换为 issues |
| 优先级三档 | 高=影响核心流程可用性，中=影响效率，低=锦上添花 |
