## 1. 平台概述

### 1.1 定位

Sisyphus-Y 是面向数据中台测试团队的 AI 辅助功能测试用例自动生成平台。平台的核心理念是显式拆分两个问题：

- **"测什么"**（M04 测试点 / 场景地图）：在进入用例生成之前，通过 AI + 人工评审确定覆盖范围
- **"怎么测"**（M05 用例步骤）：基于已确认的测试点，通过对话式 AI 生成完整的执行步骤

完整流程链路：

```
需求录入（M00）
    ↓
文档解析（M01 UDA）
    ↓
需求分析 / AI 诊断（M03）
    ↓
测试点确认 / 场景地图（M04）  ← 评审门控节点：高风险项未处理则工作台置灰
    ↓
对话式用例生成（M05）
    ↓
用例库管理（M06）
    ↓
执行结果回流（M13）
```

### 1.2 当前版本功能范围

- **已全量实现并上线前端**：M00 / M03 / M04 / M05 / M06 / M07 / M08 / M10 / M11 / M12 / M17 / M19 / M20 / M21
- **后端已实现，前端待完善**：M09（迭代测试计划）、M13（执行回流）、M16（通知）
- **已移除**：M18（协作功能，CollaborationComment 模型残留，路由未注册）

### 2.3 SSE 流式事件格式

所有流式端点（诊断/场景地图/用例生成）遵循统一格式：

```
event: thinking
data: {"delta": "..."}

event: content
data: {"delta": "..."}

event: done
data: {"usage": {...}}

event: error
data: {"message": "..."}
```

心跳：每 15 秒发送 `: keepalive`，5 分钟无输出自动关闭。

## 5. M01 文档解析引擎（UDA）

### 5.1 功能概述

UDA（Universal Document Analyzer）将上传的各类文档统一解析为结构化 Markdown，供后续 AI 模块使用。

### 5.2 支持的文档格式

| 格式 | 解析库 | 特殊处理 |
|------|-------|---------|
| `.docx` | python-docx | 提取标题层级 / 表格 / 图片 |
| `.pdf` | PyMuPDF / pypdf | 文本提取 + 图片 OCR |
| `.md` | 原生解析 | AST 类型分析 |
| 图片（PNG/JPG）| PaddleOCR | 光学字符识别 |

### 5.3 引擎层实现

- `engine/uda/ast_types.py`：文档 AST 节点类型定义
- `engine/uda/image_handler.py`：图片提取与预处理
- `engine/uda/ocr_processor.py`：PaddleOCR 调用封装
- 解析结果存储为 `requirements.raw_text`（原始），`requirements.content`（Markdown 清洗后）

### 5.4 API 端点

| 方法 | 路径 | 说明 |
|------|------|------|
| POST | `/api/uda/parse-structure` | 解析文档结构，返回 AST |
| POST | `/api/uda/parse-docx` | 解析 Word 文档 |
| POST | `/api/uda/parse-pdf` | 解析 PDF |
| POST | `/api/uda/parse-md` | 解析 Markdown |
| POST | `/api/uda/ocr-image` | 图片 OCR |

## 7. M03 需求分析（AI诊断）

### 7.1 功能概述

AI 驱动的需求健康诊断：自动扫描需求文本的缺陷（模糊表述 / 边界遗漏 / 逻辑矛盾），并通过苏格拉底式追问引导需求人员补全。

### 7.2 流程

```
触发诊断（run） → SSE 流式扫描 → AI 输出风险清单 → 前端渲染风险卡片
                                        ↓
                              用户发起追问（chat）→ SSE 流式对话
                                        ↓
                               标记风险为已确认/已忽略
                                        ↓
                               完成诊断（complete）→ 解锁场景地图生成
```

### 7.3 API 端点

| 方法 | 路径 | 说明 |
|------|------|------|
| GET | `/api/diagnosis/{req_id}` | 获取诊断报告 |
| GET | `/api/diagnosis/reports/{req_id}` | 获取报告状态 |
| POST | `/api/diagnosis/{req_id}/run` | **SSE** 触发诊断扫描 |
| POST | `/api/diagnosis/{req_id}/chat` | **SSE** 追问对话 |
| POST | `/api/diagnosis/{req_id}/create` | 创建空白报告 |
| GET | `/api/diagnosis/{req_id}/messages` | 获取对话历史 |
| POST | `/api/diagnosis/{req_id}/complete` | 标记诊断完成 |
| PATCH | `/api/diagnosis/risks/{risk_id}/confirm` | 确认单个风险 |
| PATCH | `/api/diagnosis/{req_id}/risks/{risk_id}` | 更新风险状态 |

### 7.4 前端实现

- **路由**：`/analysis`，右侧面板 "AI 分析" Tab
- **诊断启动**：点击"开始诊断"按钮，右侧 SSE 流式展示思考过程（可折叠）+ 风险清单
- **风险卡片**：按严重级别（critical / warning / info）分组，每项可追问、确认、忽略
- **StreamCursor 动画**：流式输出期间显示光标闪烁（`w-0.5 h-3.5 bg-sy-warn animate-blink`）
- **历史消息**：页面刷新后通过接口恢复，不依赖本地状态

### 7.5 引擎层实现

- `engine/diagnosis/scanner.py`：需求风险扫描器（LLM 调用，输出结构化风险列表）
- `engine/diagnosis/questioner.py`：苏格拉底式追问器
- `engine/diagnosis/checklist.py`：测试清单生成
- `engine/diagnosis/quality.py`：质量评估评分（0-100）

### 7.6 数据模型

```
diagnosis_reports
├── requirement_id (FK, 唯一)
├── status (pending/running/completed/failed)
├── overall_score (0-100)
├── completed_at

diagnosis_risks
├── report_id (FK)
├── category (completeness/logic/boundary/ambiguity)
├── severity (critical/warning/info)
├── title / description / suggestion
├── status (unconfirmed/confirmed/ignored)

diagnosis_chat_messages
├── report_id (FK)
├── role (user/assistant)
├── content
├── thinking (AI 思考过程，可能为空)
```

## 8. M04 测试点与场景地图

### 8.1 功能概述

在完成需求诊断后，AI 基于需求文本生成结构化的测试点树（场景地图），每个节点标注覆盖状态（已覆盖/AI补全/缺失/待确认）。人工评审确认后，方可进入用例生成工作台。

**门控规则**：场景地图中存在 `missing`（缺失/高风险）状态的未处理节点时，工作台"开始生成"按钮置灰。

### 8.2 API 端点

| 方法 | 路径 | 说明 |

| GET | `/api/scene-map/{req_id}` | 获取场景地图 |
| POST | `/api/scene-map/{req_id}/generate` | **SSE** 生成场景地图 |
| POST | `/api/scene-map/{req_id}/test-points` | 手动添加测试点 |
| GET | `/api/scene-map/{req_id}/test-points` | 列出所有测试点 |
| PATCH | `/api/scene-map/test-points/{id}` | 更新测试点 |
| POST | `/api/scene-map/test-points/{id}/confirm` | 确认单个测试点 |
| DELETE | `/api/scene-map/test-points/{id}` | 删除测试点 |
| POST | `/api/scene-map/{req_id}/confirm` | 整体确认场景地图 |
| POST | `/api/scene-map/{req_id}/rag-preview` | RAG 上下文预览 |
| POST | `/api/scene-map/{req_id}/test-points/batch-update` | 批量更新测试点 |
| POST | `/api/scene-map/{req_id}/test-points/reorder` | 拖拽排序测试点 |
| GET | `/api/scene-map/{req_id}/export` | 导出场景地图 |

### 8.3 前端实现

- **路由**：`/analysis`，右侧面板 "场景地图" Tab（也可从工作台 Step1 访问）
- **React Flow 可视化**：树形/思维导图布局，节点颜色编码如下：

| 状态 | 含义 | 样式 |
|------|------|------|
| `document` | 需求文档已覆盖 | `bg-sy-accent/10 border-sy-accent/35 text-sy-accent` |
| `supplemented` | AI 识别补全 | `bg-sy-warn/10 border-sy-warn/35 text-sy-warn` |
| `missing` | 缺失/高风险 | `bg-sy-danger/10 border-sy-danger text-sy-danger` |
| `pending` | 待确认 | `bg-sy-bg-3 border-dashed border-sy-border-2 text-sy-text-3` |

- **节点操作**：点击节点可编辑 / 确认 / 删除；侧边面板展示测试点详情
- **拖拽排序**：测试点列表支持 dnd-kit 拖拽重排

### 8.4 数据模型

```
scene_maps
├── requirement_id (FK, 唯一)
├── status (draft/confirmed)
├── confirmed_at

scene_nodes
├── scene_map_id (FK)
├── parent_id (自引用, 支持树形结构)
├── title / description
├── node_type (root/category/test_point)
├── coverage_status (document/supplemented/missing/pending)
├── position_x / position_y (React Flow 坐标)

test_points
├── scene_map_id (FK)
├── scene_node_id (FK)
├── title / description
├── priority (P0-P3)
├── test_type (functional/boundary/negative/performance)
├── status (draft/confirmed/ignored)
├── sort_order
```

## 9. M05 对话式用例生成工作台

### 9.1 功能概述

工作台是平台核心功能区，采用同页双步骤设计（Step 1 → Step 2，无页面跳转）：

- **Step 1：确认测试点** — 展示场景地图的已确认测试点，支持补充/修改
- **Step 2：生成用例** — 基于已确认测试点，通过 SSE 对话流式生成测试用例

平台所有核心业务对象（需求/用例/知识文档等）删除后进入回收站，30 天后自动永久清除。

### 23.2 API 端点

| 方法 | 路径 | 说明 |
|------|------|------|
| GET | `/api/recycle` | 列出回收站中的对象 |
| POST | `/api/recycle/{id}/restore` | 恢复单个对象 |
| POST | `/api/recycle/batch-restore` | 批量恢复 |
| DELETE | `/api/recycle/{id}` | 永久删除 |
| POST | `/api/recycle/clean-expired` | 清理过期对象（可手动触发）|

### 23.3 前端实现（`/recycle`）

- 对象列表：类型 / 名称 / 删除时间 / 到期倒计时
- 到期倒计时：`< 24h` 时红色警示
- 批量选择 + 恢复 / 永久删除操作

#### 数据模型

```
ai_configurations

prompt_configs
├── module (diagnosis/scene_map/generation/diff/evaluation/knowledge)
├── content (自定义 Prompt 内容)
├── version
├── is_active
```

## 25. 后端基础设施

### 25.1 数据库连接（`app/core/database.py`）

- 使用 `@lru_cache` 延迟建立连接，避免导入时连接数据库
- `AsyncSession` 通过 `Depends(get_async_session)` 注入
- 所有核心业务表通过 `SoftDeleteMixin` 实现软删除（`deleted_at` 字段）

### 25.2 Redis（`app/core/redis.py`）

- 用途：Celery broker、SSE 消息缓冲、会话缓存
- 连接池管理，支持集群模式

### 25.3 MinIO（`app/core/minio.py`）

- 存储上传文件（需求文档、知识库文档）
- 导出文件（Word/PDF/XMind）
- 预签名 URL 提供下载链接（有效期可配置）

### 25.4 Alembic 迁移

- `alembic/versions/` 记录所有 schema 变更
- 关键迁移：
  - `8f4b46d7a911`：创建 coverage_matrices / export_jobs 表
  - `d21ab83a6d2b`：创建 requirement_diffs 表
  - `ee8a72884ae2`：为 ai_configurations 添加 api_keys / vector_config JSONB 列

### 25.5 后台任务（Celery）

- `app/modules/tasks/`：Celery 任务路由
- 异步任务：PDF/Word/XMind 导出、知识库向量化、批量导入、Diff 分析

### 25.6 容错降级策略

1. 主 LLM 失败 → 重试 2 次（指数退避：1s / 2s）
2. 重试失败 → 自动降级到 `llm_fallback_provider`
3. 所有 LLM 调用设置 `trust_env=False`（绕过系统代理）
